### 🧪 Reload Window 後的「終極實戰測試」

既然您已經 Reload Window 讓 Agent 載入新工具，建議您用以下 **3 個具體的 Prompt** 來測試它的表現，看看它是否會「主動且正確地」呼叫您剛建好的 MCP 工具：

#### 測試 1：全端新增功能（測試 DTO 合約與黃金範例）
> **Prompt:** 「幫我新增一個『黑名單管理』的功能，後端需要一個 API 可以接收使用者 ID 並加入黑名單，前端需要一個 Vue 頁面可以輸入 ID 並呼叫這個 API。請確保前後端資料結構一致，並參考我們現有的程式碼風格。」
*   **預期 Agent 行為**：
    1. 呼叫 `search_code_entity("黑名單")` 確認是否已存在。
    2. 呼叫 `get_example_code("auth" 或相關模組, "Controller")` 取得後端風格。
    3. **關鍵**：在生成程式碼前，它應該在內部確立好 `BlacklistRequest { userId: string }` 的 TS Interface，並確保 Java 的 `@RequestBody` 與 Vue 的 `axios.post` payload 完全對齊。

#### 測試 2：跨模組依賴分析（測試耦合與邊界）
> **Prompt:** 「我現在要在 `order` 模組中開發一個『計算訂單總價』的 Service，但我需要用到 `product` 模組的資料。請幫我分析我應該怎麼依賴 `product` 模組才不會破壞架構？並給我這段 Service 的程式碼草稿。」
*   **預期 Agent 行為**：
    1. 呼叫 `get_module_exports("product")` 看看 `product` 模組對外暴露了哪些 public class。
    2. 呼叫 `get_module_coupling("order", "product")` 看看過去 `order` 是怎麼依賴 `product` 的。
    3. 根據回傳的邊界資訊，生成正確 import 路徑的 Service 程式碼，而不是自己瞎猜一個 `ProductService`。

#### 測試 3：模糊搜尋與除錯（測試 FTS5 效能）
> **Prompt:** 「幫我找一下專案裡有沒有跟『發送郵件』或『通知』相關的 API？我忘記是在哪個模組了，幫我列出端點和它的 Request 結構。」
*   **預期 Agent 行為**：
    1. 呼叫 `search_code_entity("郵件 通知", "api")`。
    2. 因為有 FTS5，它應該在毫秒內拿到結果。
    3. 接著自動呼叫 `get_feature_contract(endpoint)` 取得精確的 DTO 結構並展示給您。
