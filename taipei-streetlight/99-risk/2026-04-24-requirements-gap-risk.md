## 二、有 spec 但關鍵子項遺漏

### 2.1 §4.1 GIS 地圖 — 只有概念，沒有技術方案

| 遺漏項 | 需求原文摘要 | 影響 |
|--------|-------------|------|
| **坐標轉換** | TWD97 ↔ GPS ↔ 台電 TWD67 三系統互轉 | 無此功能無法匯入匯出 |
| **ArcGIS 整合** | 精確圖資用 ArcGIS 平台 | 未評估授權費用和 API |
| **OpenLayers/Google Map** | 行動端用開放圖台 | 未選型 |
| **GML 匯出** | 工務局要求的 Geography Markup Language | 完全沒提過 |
| **千分之一地形圖** | 單畫面顯示全區路燈（zoom 15） | 效能問題，16 萬個點 |
| **管線三維坐標** | 含架空管線、回路綁定 | 3D 資料結構未設計 |
| **圖資比例尺輸出** | 匯出 1/1000、A3 尺寸的向量圖面 | 需 CAD 引擎或類似工具 |
| **圖例/符號向量圖元** | 圖例需產製為電腦繪圖軟體可用之向量格式 | 未設計 |
| **分區範圍管理** | 行政區里、分隊分區、台電區處、廠商分區圖 | 多層圖層管理未規劃 |
| **管線資料庫介接** | 自工務局（道管中心）下載圖資匯入 | 外部 API 未研究 |
| **施工範圍整合** | 預定/施工中新設、建案臨拆復舊、台電施工等 | 跨系統整合未規劃 |

## 五、按優先級排序的行動建議

### P0 — 不做無法驗收

| 項目 | 理由 |
|------|------|
| §9 績效管理 | 績效扣款直接影響契約金額，機關最在意的模組 |
| §8 IoT 智能路燈 | 需求核心功能，且需外部系統串接（最長前置期） |
| §10 行動 APP | 現場人員日常工具，離線續傳是硬需求 |
| §11 儀表板 | 機關主管每天看的畫面 |
| §4.1 GIS 地圖 | 幾乎所有模組都依賴地圖顯示，是基礎設施 |

### P1 — 高度影響使用

| 項目 | 理由 |
|------|------|
| §2-(3) 臺北通/Taipeion 登入 | 機關要求的登入方式 |
| §2-(5) 密碼重設 | 基本功能缺失 |
| §5-(1) 公開報修網頁 | 民眾使用入口 |
| §5-(2) 1999 介接 | 報修來源之一 |
| §6-(1) QR Code 產生 | 燈桿號碼牌需要 |
| §2-(10) 待辦/通知提示 | 使用者登入後的核心體驗 |

### P2 — 合規與品質

| 項目 | 理由 |
|------|------|
| §2-(8) 普級合規稽核 | 資安稽核會查 |

## 六、最大風險摘要

1. **§8/§9/§10/§11 四個模組合計佔需求量的一半以上，目前完全沒有 x-plan。**
2. **§8 IoT 整合需要外部系統對接（5G 智慧杆規範、智慧電表、各區智能路燈），前置作業時間最長，建議最優先啟動技術評估。**
3. **§4.1 GIS 是橫向基礎設施**，§5-(10) 報告單繪圖、§6-(8) 預匯入地圖、§8-(2) 即時狀態地圖、§11-(2)-C 故障熱區 全部依賴它。
4. **§2-(3) 臺北通/Taipeion 登入** 涉及外部 API 對接，需提前申請和測試。
5. **§(六) 效能需求 100 人 5 秒** 是驗收硬指標，需提前規劃壓測方案。
