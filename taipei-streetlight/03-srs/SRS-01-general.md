## SRS-01-001 Web SPA + APP 雙平台交付

**來源**：§1-(1)

> 使用者端操作均能透過瀏覽器執行路燈管理平台，針對分電箱設備安全檢查、燈桿設備安全檢查等，亦能以應用程式 APP 於現場執行行動化作業。

### Use Case

| 項目 | 說明 |
|------|------|
| **Actor** | 所有角色 |
| **前置條件** | 系統已部署至生產環境 |
| **主要流程** | 1. 使用者透過瀏覽器（Chrome/Edge/Firefox/Safari）存取 Web SPA 執行管理操作 |
| | 2. 現場人員透過行動裝置 APP 執行巡查、普查、維護拍照等外勤作業 |
| | 3. Web 與 APP 共用同一後端 API，資料即時同步 |
| **後置條件** | 操作結果即時反映於 Web 與 APP 端 |

### 驗收準則

| AC ID | 驗收條件 |
|-------|---------|
| AC-01-001-1 | Web SPA 可於 Chrome、Edge、Firefox、Safari 最新版正常運作 |
| AC-01-001-2 | APP 可於 Android 12+ 與 iOS 16+ 正常運作 |
| AC-01-001-3 | Web 與 APP 使用相同 RESTful API，操作結果即時同步 |
| AC-01-001-4 | 分電箱安全檢查、燈桿安全檢查等巡檢功能在 APP 端可離線操作 |

### 技術設計

- **Web**：Vue 3 + TypeScript + Element Plus（SPA）
- **APP**：Flutter 或 React Native（待定，見 SRS-10）
- **後端**：Spring Boot 3 RESTful API，JWT 認證
- **狀態**：⚠️ Web SPA 已完成，APP 未開始

## SRS-01-002 報修時效自動追蹤與智能通報比對

**來源**：§1-(2)

> 路燈報修派工維修時效自動追蹤警示、智能路燈故障自動通報不得晚於民眾通報時間等比對功能。

### Use Case

| 項目 | 說明 |
|------|------|
| **Actor** | GOV_STAFF、GOV_MGR、CTR_ADMIN |
| **前置條件** | 報修案件已立案，或智能路燈已偵測故障 |
| **主要流程** | 1. 系統於案件立案時記錄起始時間 |
| | 2. 依案件類型自動計算預計完工期限 |
| | 3. 到期前 N 小時發送預警通知 |
| | 4. 逾期自動升級通知層級（承辦→股長→主管） |
| | 5. 智能路燈自動通報與民眾報修時間比對，標記延遲案件 |
| **例外處理** | E1: 案件退回補件時，時效自動延長 |
| | E2: 案件改分（轉送）時，重新計算時效 |
| **後置條件** | 時效紀錄完整，可做為績效計分依據 |

### 驗收準則

| AC ID | 驗收條件 |
|-------|---------|
| AC-01-002-1 | 派工維修案件自動記錄立案時間與預計完工時間 |
| AC-01-002-2 | 逾期案件自動產生警示通知，推送至對應人員 |
| AC-01-002-3 | 退回補件自動延長時效，延長天數可系統設定 |
| AC-01-002-4 | 智能路燈自動通報與最近一筆民眾報修比對，若自動通報晚於民眾報修則標記異常 |
| AC-01-002-5 | 維護時數統計可匯出做為績效評分依據 |

### 資料模型

- `repair_tickets.created_at`：立案時間
- `repair_tickets.deadline_at`：預計完工時限
- `repair_tickets.completed_at`：實際完工時間
- `system_settings.repair_deadline_hours`：維修時限（小時），可系統設定
- `fault_tickets.auto_report_at`：智能路燈自動通報時間
- `repair_tickets.citizen_report_at`：民眾報修時間

### 狀態：⚠️ 時效欄位已存在，追蹤與比對邏輯待實作

## SRS-01-003 GIS 圖資校修與財產系統介接

**來源**：§1-(3)

> 結合本府「臺北市政府財產管理系統」、工務局管線資料庫等，校修完備機關路燈資產資訊，確保資料及格式之正確性及一致性。

### Use Case

| 項目 | 說明 |
|------|------|
| **Actor** | GOV_STAFF、GOV_ADMIN |
| **前置條件** | 已取得外部系統介接授權 |
| **主要流程** | 1. 定期或手動觸發與「財產管理系統」資料同步 |
| | 2. 匯入工務局管線資料庫 GML 圖資 |
| | 3. 比對本地資產 vs 外部資產，列出差異 |
| | 4. 承辦人確認差異後，批次更新或逐筆校正 |
| **例外處理** | E1: 外部系統不可用時，記錄同步失敗並排程重試 |
| **後置條件** | 資產資料與外部系統一致 |

### 驗收準則

| AC ID | 驗收條件 |
|-------|---------|
| AC-01-003-1 | 可匯入工務局 GML 格式管線圖資 |
| AC-01-003-2 | 匯入前自動比對差異，需人工確認後才更新 |
| AC-01-003-3 | 與財產管理系統資料同步（批次或 API），欄位格式對應明確 |
| AC-01-003-4 | 同步失敗時產生異常紀錄，可查詢排查 |

### 外部介接

| 外部系統 | 介接方式 | 狀態 |
|---------|---------|------|
| 財產管理系統 | 待確認（批次/API） | EXT-08，待協調 |
| 工務局管線資料庫 | GML Import | EXT-06，待申請 |

## SRS-01-004 照片影片儲存、批次匯出與 AI 影像辨識

**來源**：§1-(4)

> 照片、影片儲存於平台資料庫；可依條件批次匯出至外部暫存空間；提供 AI 影像辨識引擎自動解析分類。

### Use Case

**UC-01-004-A 媒體儲存與匯出**

| 項目 | 說明 |
|------|------|
| **Actor** | GOV_ADMIN、GOV_STAFF |
| **主要流程** | 1. 所有維護/巡查/換裝照片上傳時自動儲存至平台 |
| | 2. 管理員依條件篩選（時間範圍、區域、地點）批次匯出 |
| | 3. 匯出至指定外部暫存空間（FTP/NAS/雲端） |
| | 4. 可設定規則自動將 N 年前舊照片備份後移出主資料庫 |

**UC-01-004-B AI 影像辨識**

| 項目 | 說明 |
|------|------|
| **Actor** | GOV_STAFF |
| **主要流程** | 1. 使用者指定條件（地點、相似性）觸發 AI 分析 |
| | 2. AI 引擎自動分類照片、找出相似或重複影像 |
| | 3. 結果以標籤、群組方式呈現供人工確認 |

### 驗收準則

| AC ID | 驗收條件 |
|-------|---------|
| AC-01-004-1 | 上傳照片/影片自動儲存，具備時間、GPS、案件關聯 metadata |
| AC-01-004-2 | 可依時間、區域、地點批次篩選匯出照片，支援 ZIP 打包下載 |
| AC-01-004-3 | 可設定自動備份規則，將指定時間前的媒體移至備份儲存 |
| AC-01-004-4 | AI 引擎可自動分類照片並找出相似/重複影像，準確率 ≥ 85% |
| AC-01-004-5 | AI 引擎具合法授權，由得標廠商提供 |

### 外部介接

| 外部系統 | 介接方式 | 狀態 |
|---------|---------|------|
| AI 影像辨識引擎 | REST API | EXT-12，待選型 |

### 狀態：❌ 未開始

## SRS-01-005 動態欄位擴充

**來源**：§1-(5)

> 需考量未來增設各類資產設備及資料欄位之需求，可直接於介面操作新增不需額外開發，且各子系統間之數據資料需維持其完整一致性。

### Use Case

| 項目 | 說明 |
|------|------|
| **Actor** | GOV_ADMIN |
| **前置條件** | 具有系統管理權限 |
| **主要流程** | 1. 管理員於設備類型管理介面新增/編輯自訂欄位 |
| | 2. 定義欄位名稱、資料型態（文字/數值/日期/下拉選項）、是否必填 |
| | 3. 新欄位即時出現在對應設備的新增/編輯/查詢表單 |
| | 4. 資料存入 JSONB `attributes` 欄位 |
| **後置條件** | 新欄位可搜尋、匯出，跨子系統一致 |

### 驗收準則

| AC ID | 驗收條件 |
|-------|---------|
| AC-01-005-1 | 管理員可於介面新增自訂欄位，無需修改程式碼或重新部署 |
| AC-01-005-2 | 自訂欄位支援至少 4 種資料型態（文字、數值、日期、選項） |
| AC-01-005-3 | 新增欄位即時反映於新增/編輯/查詢/匯出功能 |
| AC-01-005-4 | 各子系統使用相同設備資料時，自訂欄位一致 |

### 技術設計

- **已實作**：`devices.attributes` JSONB 欄位，支援各設備類型的 type-specific 屬性
- **各類型 JSONB Schema**：POLE（9 欄位）、LUMINAIRE（12 欄位）、PANEL_BOX（7 欄位）、CONTROLLER（7 欄位）、ATTACHMENT（5 欄位）
- **待實作**：前端動態表單產生器（Schema-driven form rendering）

### 狀態：✅ JSONB 模型已建置，⚠️ 介面動態新增欄位待開發
