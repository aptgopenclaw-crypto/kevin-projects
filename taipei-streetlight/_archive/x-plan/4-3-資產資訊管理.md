### 1. 後端：資料表設計

**Flyway Migration** (`V30__device__create_tables.sql`)

#### 1-1 設備主表（devices）

```sql
-- ============================================================
-- 設備主表：所有 IoT 設備共用，依 device_type 區分類型
-- ============================================================
CREATE TABLE devices (
    id                  BIGSERIAL       PRIMARY KEY,
    tenant_id           VARCHAR(50)     NOT NULL REFERENCES tenant(tenant_id),
    device_type         VARCHAR(30)     NOT NULL,
    device_code         VARCHAR(100)    NOT NULL,
    device_name         VARCHAR(200),

    -- 坐標（三套）
    twd97_x             NUMERIC(12,3),
    twd97_y             NUMERIC(12,3),
    lng                 NUMERIC(11,7),
    lat                 NUMERIC(10,7),
    elevation           NUMERIC(8,3),
    taipower_coord      VARCHAR(100),

    -- 組織歸屬
    dept_id             BIGINT          REFERENCES dept_info(dept_id),
    contract_id         BIGINT,
    property_owner      VARCHAR(200),

    -- 設備狀態
    status              VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE',
    installed_at        DATE,
    decommissioned_at   DATE,

    -- 連線拓撲
    parent_device_id    BIGINT          REFERENCES devices(id),
    connectivity_type   VARCHAR(20)     DEFAULT 'NONE',
    network_config      JSONB           DEFAULT '{}',
    last_heartbeat_at   TIMESTAMP,

    -- 電力回路
    circuit_id          BIGINT,

    -- 專有欄位（依 device_type 不同）
    attributes          JSONB           DEFAULT '{}',

    -- 審計
    created_by          VARCHAR(50),
    created_at          TIMESTAMP       NOT NULL DEFAULT now(),
    updated_at          TIMESTAMP       NOT NULL DEFAULT now()
);

CREATE INDEX idx_devices_tenant ON devices(tenant_id);
CREATE INDEX idx_devices_type ON devices(tenant_id, device_type);
CREATE INDEX idx_devices_code ON devices(tenant_id, device_code);
CREATE INDEX idx_devices_parent ON devices(parent_device_id);
CREATE INDEX idx_devices_circuit ON devices(circuit_id);
CREATE INDEX idx_devices_status ON devices(tenant_id, status);
CREATE INDEX idx_devices_dept ON devices(dept_id);
CREATE INDEX idx_devices_coords ON devices(lng, lat);
CREATE INDEX idx_devices_attrs ON devices USING GIN (attributes);

COMMENT ON TABLE devices IS '設備主表：燈桿、燈具、分電箱、控制器、附掛物等';
COMMENT ON COLUMN devices.device_type IS 'POLE / LUMINAIRE / PANEL_BOX / CONTROLLER / POWER_EQUIPMENT / ATTACHMENT';
COMMENT ON COLUMN devices.connectivity_type IS 'NONE / DIRECT / GATEWAY';
COMMENT ON COLUMN devices.attributes IS '依 device_type 存放專有欄位（JSONB）';
```

#### 1-2 電力回路表（circuits）

```sql
-- ============================================================
-- 電力回路：分電箱下的回路，與通訊拓撲獨立
-- ============================================================
CREATE TABLE circuits (
    id                      BIGSERIAL       PRIMARY KEY,
    tenant_id               VARCHAR(50)     NOT NULL REFERENCES tenant(tenant_id),
    panel_box_device_id     BIGINT          REFERENCES devices(id),
    circuit_number          VARCHAR(50)     NOT NULL,

    taipower_account        VARCHAR(50),
    usage_type              VARCHAR(50),
    status                  VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE',
    created_at              TIMESTAMP       NOT NULL DEFAULT now(),
    updated_at              TIMESTAMP       NOT NULL DEFAULT now(),

    UNIQUE(tenant_id, circuit_number)
);

COMMENT ON COLUMN circuits.panel_box_device_id IS
  'Nullable：電池供電 IoT 或未知分電箱時為 NULL，有資料時自動關聯';

ALTER TABLE devices ADD CONSTRAINT fk_devices_circuit
    FOREIGN KEY (circuit_id) REFERENCES circuits(id);

CREATE INDEX idx_circuits_tenant ON circuits(tenant_id);
CREATE INDEX idx_circuits_panel ON circuits(panel_box_device_id);

COMMENT ON TABLE circuits IS '電力回路：一個分電箱可含多條回路，一條回路控制多盞燈';
```

#### 2-3 設備狀態（status）

| 值 | 說明 | 觸發方式 |
|------|------|----------|
| `ACTIVE` | 正常 | 預設 / E9 結案審核通過 |
| `REPORTED` | 已報修 | E1 障礙申告審核通過（自動） |
| `UNDER_REPAIR` | 維修中 | E4 報修派工（自動） |

| `DECOMMISSIONED` | 已除帳/報廢 | 手動操作（不可逆） |

> REPORTED → UNDER_REPAIR → ACTIVE 由工作流事件自動驅動，前端用戶不可手動設定。
> 詳細狀態生命週期圖見 [phase-1-foundation.md](phase-1-foundation.md) §1-2a。

### 3. 受眾範圍與 DataScope 機制

#### 3-1 設備查詢的 DataScope 適用性

設備管理與公告不同，適合使用現有 `@DataPermission` + `DataScopeHelper` AOP 機制：

| 角色 | DataScope | 可見設備範圍 |
|------|-----------|-------------|
| ADMIN / SUPER_ADMIN | `ALL` | 租戶內所有設備 |
| DEPT_ADMIN | `THIS_LEVEL_AND_BELOW` | 自己部門及下屬部門管轄的設備 |
| DEPT_USER / FIELD_USER | `THIS_LEVEL` | 自己部門管轄的設備 |

### 8. 邊界情境處理

#### 8-1 設備編號唯一性

同租戶 + 同設備類型下，`device_code` 必須唯一。後端 Service 層在新增/編輯時檢查，衝突時拋出 `BusinessException(ErrorCode.DEVICE_CODE_DUPLICATE)`。

#### 8-2 設備刪除（軟刪除）

刪除設備不做物理刪除，改為 `status = 'DECOMMISSIONED'` + 記錄 `decommissioned_at`。已除帳設備在一般查詢中不顯示，管理員可透過狀態篩選查看。

刪除前檢查：
- 若設備有子設備（`parent_device_id` 指向此設備），需先解除或移轉子設備
- 若設備在進行中的障礙工單中，需先處理工單

#### 8-3 拓撲循環防護

新增/編輯 `parent_device_id` 時，後端用 recursive CTE 檢查是否形成循環：

```sql
WITH RECURSIVE ancestors AS (
    SELECT id, parent_device_id FROM devices WHERE id = :parentId
    UNION ALL
    SELECT d.id, d.parent_device_id FROM devices d JOIN ancestors a ON d.id = a.parent_device_id
)
SELECT 1 FROM ancestors WHERE id = :childId
```

若查到結果，拒絕操作並回傳 `BusinessException(ErrorCode.DEVICE_CIRCULAR_REFERENCE)`。

#### 8-4 坐標轉換

前端提供 TWD97 ↔ GPS 互轉功能（JavaScript 端實作）。台電坐標（TWD67 TM2）轉換需額外參數表，第一版手動填寫，後續版本支援自動轉換。

#### 8-5 JSONB attributes 驗證

後端不對 attributes 做嚴格 schema 驗證（保持彈性），但提供以下防護：
- 最大大小限制（如 10KB）
- 不接受巢狀超過 3 層的 JSON
- 前端依 `device_type` 動態渲染對應欄位表單，引導正確輸入

#### 8-6 雙路徑障礙偵測機制

IoT 設備同時依賴電力路徑與通訊路徑，兩條路徑的故障偵測邏輯不同：

| | 電力路徑 | 通訊路徑 |
|---|---|---|
| 上游 | 分電箱 → 回路 | Gateway → SIM → 雲端 |
| 故障表現 | 燈不亮（物理滅燈） | 設備離線（telemetry 中斷，**燈可能還亮**） |
| 使用者感知 | 可感知（民眾 / 巡檢發現） | **不可感知** — 只有系統監控能偵測 |
| 偵測方式 | 被動（報修驅動） | 主動（心跳掃描） |
| 根因 | 回路跳電、線路斷 | Gateway 當機、SIM 到期、ISP 斷線 |
| 復原 | 來電即恢復 | Gateway 恢復後批次 backfill |

偵測觸發時機分為三層：

**第一層：被動偵測（報修驅動）**
1. 新工單建立後，同步檢查同 `circuit_id` 近 30 分鐘內的其他 OPEN 工單 → 觸發回路級關聯
2. 同時檢查同 `parent_device_id` 下是否已存在 Gateway 離線告警 → 合併至現有 correlation

**第二層：主動偵測（心跳掃描，每 5 分鐘）**
1. 掃描所有 `connectivity_type = 'DIRECT'` 且具備 Gateway 角色的設備（`attributes->>'is_gateway' = true`）
2. 若 Gateway 本體 `last_heartbeat_at` 超過閾值（15 分鐘）→ 標記 OFFLINE + 查詢子設備
3. 若同一 Gateway 下 ≥ 50% 子設備心跳超時 → 建立 `fault_correlation(root_cause_type='GATEWAY')` + AUTO_ALERT 工單
4. 推播通知：「Gateway PB-A01 離線，影響 N 台設備，請檢查 SIM / 網路」

**第三層：預防性偵測（每日 09:00）**
1. 掃描 `network_config->>'sim_expiry'` 欄位
2. SIM 到期日 ≤ 30 天 → 推播預警通知（不建工單）
3. SIM 到期日 ≤ 7 天 → 推播緊急通知 + 建立預防性工單

**Gateway 復原流程**：
- Gateway 恢復心跳 → status = ONLINE
- 子設備陸續回報 → 處理 backfill 資料（離線期間快取的 telemetry）
- 所有子設備恢復 → correlation status = 'RESOLVED'，自動結案

#### 8-7 批次匯入設備

第一版不支援批次匯入（需求 4-2 E 提到「清冊匯入」），預留 API 端點但不實作：
- `POST /v1/auth/devices/import`（返回 501 Not Implemented）
- 後續版本支援 CSV/XLS 批次匯入 + 差異比對

#### 8-8 分電箱高度檢查

需求明確提及分電箱裝置高度「是否達 1.5 公尺」。`PANEL_BOX` 的 attributes 中 `mount_height` 欄位為數值，`meets_height_req` 為 boolean。前端新增分電箱時，輸入高度後自動計算並提示。

#### 8-9 設備照片管理

設備相關照片（換裝照、報修照、巡檢照）統一存放在 `device_events.attachments` JSONB 欄位中：
```json
[
  {"url": "/files/devices/123/photo1.jpg", "desc": "換裝前", "uploaded_at": "2026-04-20"},
  {"url": "/files/devices/123/photo2.jpg", "desc": "換裝後", "uploaded_at": "2026-04-20"}
]
```
第一版照片上傳至本機檔案系統（`/uploads/devices/`），後續可遷移至 S3 / MinIO。

#### 8-10 回路刪除限制

刪除回路前檢查是否仍有設備綁定（`devices.circuit_id = ?`），若有則拒絕刪除並回傳 `BusinessException(ErrorCode.CIRCUIT_HAS_DEVICES)`。

#### 8-11 設備搜尋與篩選

管理頁面提供以下篩選：
- **設備類型**：下拉選單（全部 / 燈桿 / 燈具 / 分電箱 / ...）
- **狀態**：使用中 / 暫停 / 已除帳
- **部門**：部門下拉（受 DataScope 限制）
- **關鍵字**：模糊比對設備編號 + 設備名稱
- **上層設備**：指定 Gateway 查看其下掛設備
- **回路**：指定回路查看其下設備

#### 8-12 設備統計 Dashboard

`GET /devices/stats` 回傳概覽數據，供前端 Dashboard 卡片或圖表使用：
```json
{
  "totalDevices": 5200,
  "byType": { "POLE": 2100, "LUMINAIRE": 2100, "PANEL_BOX": 80, "CONTROLLER": 900, "ATTACHMENT": 20 },
  "byStatus": { "ACTIVE": 4800, "REPORTED": 50, "UNDER_REPAIR": 100, "INACTIVE": 150, "DECOMMISSIONED": 50 },
  "onlineRate": 0.95,
  "openFaults": 12
}
```

### D-6 Graceful Degradation（漸進式資料支援）

> 核心原則：**有資料就啟用，沒資料就跳過，不 crash 不報錯**

IoT 設備形態多元（電池供電/市電、自帶 SIM/Gateway 中繼、有/無回路），系統不假定所有欄位都有值。

#### Nullable 策略

| 欄位 | Nullable | 無資料時行為 |
|------|----------|-------------|
| `devices.parent_device_id` | ✅ | 視為頂層設備，拓撲檢視僅顯示平面列表 |
| `devices.circuit_id` | ✅ | 無回路歸屬，回路維度的關聯偵測自動跳過 |
| `devices.contract_id` | ✅ | 無契約，保固到期提醒不觸發 |
| `devices.dept_id` | ✅ | 無部門歸屬（不推薦，但不阻擋建立） |
| `devices.connectivity_type` | DEFAULT 'NONE' | 無通訊能力，Gateway 心跳偵測跳過此設備 |
| `devices.network_config` | DEFAULT '{}' | 空物件，SIM 到期掃描跳過 |
| `devices.last_heartbeat_at` | ✅ | 無心跳紀錄，不納入離線偵測 |
| `circuits.panel_box_device_id` | ✅ | 回路可獨立存在（不知道分電箱在哪） |

#### 功能退化矩陣

| 場景 | 設備 CRUD | 地圖 | 障礙工單 | 拓撲 | 回路關聯偵測 | GW 心跳偵測 | SIM 到期 |
|------|-----------|------|----------|------|-------------|------------|---------|
| 只有路燈（LUMINAIRE） | ✅ | ✅ | ✅ | 平面 | ❌ | ❌ | ❌ |
| 路燈 + 回路編號 | ✅ | ✅ | ✅ | 平面 | ✅ | ❌ | ❌ |
| 路燈 + Gateway | ✅ | ✅ | ✅ | ✅樹狀 | ❌ | ✅ | 視 SIM |
| 路燈 + 回路 + Gateway | ✅ | ✅ | ✅ | ✅樹狀 | ✅ | ✅ | ✅ |
| 電池 IoT（無電力箱） | ✅ | ✅ | ✅ | 平面 | ❌ | 視連線 | 視 SIM |

#### Service 層防禦寫法

```java
// FaultCorrelationService — 回路偵測前先檢查
if (ticket.getCircuitId() != null) {
    // 有回路 → 執行回路維度關聯偵測
}

// FaultCorrelationService — Gateway 偵測前先檢查
if (device.getConnectivityType() == ConnectivityType.GATEWAY
        && device.getParentDeviceId() != null) {
    // 有 Gateway → 檢查 parent 心跳
}

// SIM 到期掃描 — 只掃有 SIM 的設備
@Query("SELECT d FROM Device d WHERE d.connectivityType = 'DIRECT' "
     + "AND d.networkConfig IS NOT NULL "
     + "AND jsonb_extract_path_text(d.networkConfig, 'sim_expiry') IS NOT NULL")
List<Device> findDevicesWithSim();
```

> **device_type enum 保留全部值**（POLE / LUMINAIRE / PANEL_BOX / CONTROLLER / POWER_EQUIPMENT / ATTACHMENT）。只有路燈資料時其他 type 不會出現在表中，不佔成本；未來若市府提供更多設備資料可直接匯入，零 DDL 變更。
