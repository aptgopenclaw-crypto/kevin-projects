### 1. 後端：公告資料表

**Flyway Migration** (`V29__announcement__create_tables.sql`)

```sql
CREATE TABLE announcements (
    id              BIGSERIAL    PRIMARY KEY,
    tenant_id       VARCHAR(50)  NOT NULL REFERENCES tenant(tenant_id),
    title           VARCHAR(200) NOT NULL,
    content         TEXT         NOT NULL,
    status          VARCHAR(20)  NOT NULL DEFAULT 'DRAFT',
    scope           VARCHAR(20)  NOT NULL DEFAULT 'ALL',
    pinned          BOOLEAN      NOT NULL DEFAULT false,
    publish_at      TIMESTAMP,
    expire_at       TIMESTAMP,
    created_by      VARCHAR(50),
    created_by_name VARCHAR(100),
    created_at      TIMESTAMP    NOT NULL DEFAULT now(),
    updated_at      TIMESTAMP    NOT NULL DEFAULT now()
);

CREATE INDEX idx_announcements_tenant_id ON announcements(tenant_id);
CREATE INDEX idx_announcements_status ON announcements(tenant_id, status, publish_at DESC);

-- 公告目標部門（多對多）：scope=DEPT 時指定一或多個目標部門
CREATE TABLE announcement_depts (
    announcement_id BIGINT NOT NULL REFERENCES announcements(id) ON DELETE CASCADE,
    dept_id         BIGINT NOT NULL REFERENCES dept_info(dept_id),
    PRIMARY KEY (announcement_id, dept_id)
);

CREATE INDEX idx_announcement_depts_dept ON announcement_depts(dept_id);

-- 已讀追蹤表
CREATE TABLE announcement_reads (
    id              BIGSERIAL   PRIMARY KEY,
    announcement_id BIGINT      NOT NULL REFERENCES announcements(id) ON DELETE CASCADE,
    user_id         VARCHAR(50) NOT NULL,
    read_at         TIMESTAMP   NOT NULL DEFAULT now(),
    UNIQUE (announcement_id, user_id)
);

CREATE INDEX idx_announcement_reads_user ON announcement_reads(user_id);
```

**多租戶設計**：與 `system_settings` 相同 pattern，Entity 使用 `@Filter("tenantFilter")` + `TenantAware` + `TenantEntityListener`，Repository 繼承 `TenantScopedRepository`。

**欄位設計**

| 欄位 | 型別 | 說明 |
|------|------|------|
| `id` | BIGSERIAL | 主鍵 |
| `tenant_id` | VARCHAR(50) | 租戶 ID（FK → tenant） |
| `title` | VARCHAR(200) | 公告標題 |
| `content` | TEXT | 公告內容（第一版純文字） |
| `status` | VARCHAR(20) | 狀態：`DRAFT`（草稿）、`PUBLISHED`（已發佈） |
| `scope` | VARCHAR(20) | 受眾範圍：`ALL`（全體）、`DEPT`（指定部門） |
| `pinned` | BOOLEAN | 是否置頂 |
| `publish_at` | TIMESTAMP | 發佈時間（排序用，支援排程發佈） |
| `expire_at` | TIMESTAMP | 過期時間（預設 publish_at + 30 天，勾選「永不過期」時為 NULL） |
| `created_by` | VARCHAR(50) | 建立者 user_id（用於權限比對，如 `u-tpe-admin`） |
| `created_by_name` | VARCHAR(100) | 建立者顯示名稱（用於前台顯示，如「系統管理員」） |
| `created_at` | TIMESTAMP | 建立時間 |
| `updated_at` | TIMESTAMP | 更新時間 |

**受眾範圍（scope）設計**

| scope | announcement_depts | 意義 |
|-------|-------------------|------|
| `ALL` | 無記錄 | 租戶內全體使用者可見 |
| `DEPT` | 一或多筆部門 ID | 僅指定部門成員可見 |

**狀態設計**

```
DRAFT（草稿）
  ↓ 發佈（立即或排程）
PUBLISHED（已發佈）
  ↓ 退回草稿
DRAFT
```

前台可見條件：
```sql
status = 'PUBLISHED'
  AND publish_at <= now()
  AND (expire_at IS NULL OR expire_at > now())
  AND (
    scope = 'ALL'
    OR EXISTS (
      SELECT 1 FROM announcement_depts ad
      WHERE ad.announcement_id = a.id AND ad.dept_id = :userDeptId
    )
  )
```

**API 端點**

| Method | Path | 權限 | 說明 |
|--------|------|------|------|
| `GET` | `/v1/auth/announcements` | 已登入 | 查詢公告（自動過濾受眾範圍 + 時效） |
| `GET` | `/v1/auth/announcements/{id}` | 已登入 | 取得單筆公告詳情 |
| `GET` | `/v1/auth/announcements/unread-count` | 已登入 | 取得未讀公告數量 |
| `POST` | `/v1/auth/announcements/{id}/read` | 已登入 | 標記某則公告為已讀 |
| `POST` | `/v1/auth/announcements/read-all` | 已登入 | 全部標為已讀 |
| `POST` | `/v1/auth/announcements` | `ANNOUNCEMENT_MANAGE` | 新增公告 |
| `PUT` | `/v1/auth/announcements/{id}` | `ANNOUNCEMENT_MANAGE` | 編輯公告 |
| `DELETE` | `/v1/auth/announcements/{id}` | `ANNOUNCEMENT_MANAGE` | 刪除公告 |

**查詢 API 說明**：
- 一般使用者：自動過濾 `status=PUBLISHED` + 時效 + 受眾（`scope=ALL` 或使用者部門在 `announcement_depts` 中）
- 管理員帶 `?admin=true`：回傳所有狀態（含 DRAFT、已過期），用於管理頁面。**後端必須驗證呼叫者具有 `ANNOUNCEMENT_MANAGE` 權限才套用 admin 模式，否則忽略該參數**
  - **ADMIN / SUPER_ADMIN**：看到所有公告（不限 `created_by`）
  - **DEPT_ADMIN**：看到「自己建立的公告」＋「他人發佈且受眾包含自己部門的公告」。後者為唯讀（不可編輯/刪除）
- 分頁：`page`、`size`（預設 page=0, size=10）
- 排序：`pinned DESC, publish_at DESC`（置頂優先，再按發佈時間倒序）
- Response 含 `isRead: boolean` 欄位（後端 LEFT JOIN `announcement_reads` 判定），供 Bell 與列表頁顯示已讀/未讀標記

### 2. 受眾範圍與 DataScope 機制

#### 2-1 為什麼不直接套用 DataScope AOP

現有 `@DataPermission` + `DataScopeHelper` 機制是**讀取端過濾**（依查詢者角色決定可見範圍），適用於 User / Dept 管理。

公告的受眾控制是**發佈端指定**（發佈者選擇「這則公告給誰看」），語意方向不同：

| | DataScope（User/Dept 查詢） | 公告受眾 |
|---|---|---|
| 過濾依據 | 查詢者的角色 DataScope | 公告自身的 `scope` + `announcement_depts` |
| 語意 | 「我能看到哪些部門的資料」 | 「這則公告要給誰看」 |
| 方向 | 讀取端動態過濾 | 發佈端靜態指定 |

**結論**：不套用 `@DataPermission` AOP，改用明確的 `scope` + `announcement_depts` junction table。

#### 2-2 復用 DataScopeHelper 限制發佈範圍

雖然不用 AOP，但可**復用 `DataScopeHelper.getVisibleDeptIds()`** 來限制發佈者可選的部門範圍：

| 發佈者角色 | DataScope | 可設 scope | 可選部門（多選） |
|-----------|-----------|-----------|-------------------|
| ADMIN / SUPER_ADMIN | `ALL` | `ALL` 或 `DEPT` | 任意部門（多選，不限） |
| DEPT_ADMIN | `THIS_LEVEL` | 僅 `DEPT` | 僅自己所屬部門（自動填入，不可更改） |
| DEPT_ADMIN | `THIS_LEVEL_AND_BELOW` | 僅 `DEPT` | 僅自己所屬部門（自動填入，不可更改） |

**後端驗證**：`AnnouncementService.create()` 中檢查：
```java
if (user.getDataScope() != "ALL") {
    // DEPT_ADMIN：強制 scope = DEPT
    // 強制 targetDeptIds = [自己的部門 ID]，不接受前端傳入的值
} else {
    // ADMIN：scope=DEPT 時檢查 targetDeptIds 不為空
}
```

**行為說明**：

| 元素 | 行為 |
|------|------|
| 🔔 badge 數字 | 顯示未讀公告數量，為 0 時隱藏 badge |
| 🔵 藍點 | 標記未讀公告 |
| 點擊某則公告 | 導航至公告列表頁 `/announcements`（在列表頁點擊展開時才標記已讀） |

| 「查看所有公告 →」 | 跳轉公告列表頁（分頁瀏覽所有公告，含已讀/未讀標記） |
| Popover 顯示筆數 | 最新 10 筆（含已讀/未讀） |

**未讀數量 fetch 時機**：
- 登入成功時立即 fetch
- 之後每 5 分鐘定時輪詢（`setInterval`）
- 路由切換時不額外 fetch（避免過多請求）

**未讀數量查詢 SQL**：
```sql
SELECT COUNT(*) FROM announcements a
WHERE a.tenant_id = :tenantId
  AND a.status = 'PUBLISHED'
  AND a.publish_at <= now()
  AND (a.expire_at IS NULL OR a.expire_at > now())
  AND (a.scope = 'ALL' OR EXISTS (
    SELECT 1 FROM announcement_depts ad
    WHERE ad.announcement_id = a.id AND ad.dept_id = :userDeptId
  ))
  AND NOT EXISTS (
    SELECT 1 FROM announcement_reads r
    WHERE r.announcement_id = a.id AND r.user_id = :userId
  )
```

**公告列表頁**（「查看所有公告」跳轉目的地）：
- 路徑：`/announcements`（一般使用者可見，非 admin 頁面）
- 分頁顯示所有可見的已發佈公告（含已讀/未讀標記）
- 點擊標題展開內容 + 自動標記已讀
- 每頁 10 筆，支援翻頁瀏覽歷史公告

**與首頁公告卡片的關係**：

| 元件 | 位置 | 用途 |
|------|------|------|
| Header 🔔 Bell | 全站 header，任何頁面可見 | 輕量提示 + 快速瀏覽最新 10 筆 |
| 公告列表頁 | `/announcements` | 完整瀏覽所有公告，分頁，含已讀標記 |

兩者互補：Bell 是**即時提醒 + 快速一覽**，列表頁是**完整查閱**。Dashboard 保留給路燈狀態、地圖、統計等核心業務資訊。

**欄位驗證**：
- 標題：必填，最長 200 字元
- 內容：必填
- 受眾：ADMIN 可選「全體」或「指定部門」（多選）；DEPT_ADMIN 自動鎖定為自己部門，不可更改
- 部門多選：ADMIN 顯示所有部門（`el-select multiple`）；DEPT_ADMIN 自動填入自己部門且不可編輯
- 狀態：預設 DRAFT
- 發佈時間：選「立即」→ `publish_at = now()`；選「排程」→ DateTimePicker 選擇未來時間
- 過期時間：預設帶入 `publish_at + 30天`；切換 publish_at 時自動連動更新，**但一旦手動修改過 expire_at 則不再自動連動**（前端用 `expireManuallyEdited` flag 追蹤）；勾選「永不過期」→ `expire_at = NULL`；需 > publish_at

### 8. 邊界情境處理

#### 8-1 公告過期自動隱藏

新增公告時 `expire_at` 預設為 `publish_at + 30 天`，確保多數公告自動到期隱藏，避免長期累積。管理員可手動調整過期時間，或勾選「永不過期」（`expire_at = NULL`）用於重大政策公告。

過期的公告不會在前台 API 回傳，但仍保留在資料庫中。管理員在管理頁面仍可看到所有公告（含已過期）。

**方案**：Service 層查詢條件過濾，不使用排程刪除
```sql
WHERE status = 'PUBLISHED'
  AND publish_at <= now()
  AND (expire_at IS NULL OR expire_at > now())
  AND (
    scope = 'ALL'
    OR EXISTS (
      SELECT 1 FROM announcement_depts ad
      WHERE ad.announcement_id = a.id AND ad.dept_id = :userDeptId
    )
  )
```

#### 8-2 排程發佈（不需後端排程器）

管理員設定 `status=PUBLISHED` + `publish_at` 為未來時間時，公告自然不會在前台顯示（因查詢條件 `publish_at <= now()`），直到時間到達才自動出現。**純靠 SQL 查詢條件實現，不需要 Quartz / @Scheduled 等排程器**。

#### 8-3 置頂排序

多筆置頂公告時，按 `publish_at DESC` 排序（最新的置頂排最前）。非置頂公告依 `publish_at DESC` 排在置頂之後。

```sql
ORDER BY pinned DESC, publish_at DESC
```

#### 8-4 草稿可見性

- 一般使用者：只看到 `PUBLISHED` 且未過期且在受眾範圍內的公告
- 管理員（管理頁面 `?admin=true`）：看到所有狀態，包含 `DRAFT` 和已過期

#### 8-5 DEPT_ADMIN 發佈範圍限制

DEPT_ADMIN 只能發佈 `scope=DEPT` 的公告，且**只能公告給自己所屬部門**，不能選擇其他部門：
- 後端強制 `targetDeptIds = [自己的 dept_id]`，忽略前端傳入的部門選擇
- 前端 Dialog 自動填入自己部門且不可編輯（disabled）

#### 8-6 公告內容格式

第一版使用純文字（`el-input type="textarea"`），不引入富文本編輯器。
若未來需要格式化內容，可升級為 Markdown 或整合 TinyMCE / Quill 等編輯器。

#### 8-7 無公告空狀態

若 Bell Popover 或公告列表頁沒有任何可見的已發佈公告，顯示「目前沒有公告」提示文字。

#### 8-8 刪除確認

刪除公告前彈出 `ElMessageBox.confirm` 確認，避免誤刪已發佈公告。

#### 8-9 DEPT_ADMIN 管理頁面可見範圍與操作權限

DEPT_ADMIN 在管理頁面可看到：
1. **自己建立的公告**（`created_by` = 當前 user_id）→ 可編輯、可刪除
2. **他人發佈且受眾包含自己部門的公告**（`scope='DEPT'` 且 `announcement_depts` 含自己 dept_id）→ **唯讀**，不可編輯/刪除

後端查詢條件（DEPT_ADMIN + `admin=true`）：
```sql
WHERE (a.created_by = :userId
       OR (a.scope = 'DEPT' AND EXISTS (
           SELECT 1 FROM announcement_depts ad
           WHERE ad.announcement_id = a.id AND ad.dept_id = :userDeptId)))
```

前端操作按鈕顯示規則：Response DTO 的 `editable` 欄位為 `true` 時才顯示 ✏️ 🗑 按鈕。

ADMIN / SUPER_ADMIN 不受此限，可看到並操作所有公告。

#### 8-10 已讀記錄與公告刪除

`announcement_reads` 設有 `ON DELETE CASCADE`，公告刪除時已讀記錄自動清除，不會產生孤兒資料。

#### 8-11 未讀數量輪詢頻率

每 5 分鐘輪詢一次 `GET /announcements/unread-count`。此 API 查詢量輕（COUNT + EXISTS），不會對資料庫造成壓力。若未來需要更即時的通知，可升級為 WebSocket / SSE。

#### 8-12 公告列表頁翻頁

「查看所有公告」頁面使用與管理頁面相同的分頁機制（`page` + `size`），但只顯示 PUBLISHED 且在受眾範圍內的公告。每頁 10 筆，使用者可翻頁瀏覽歷史公告。每則公告旁標記已讀/未讀狀態（藍點），點擊展開即標記已讀。

#### 8-13 多部門受眾設計

`scope=DEPT` 時透過 `announcement_depts` junction table 支援一或多個目標部門。ADMIN 可多選任意部門，DEPT_ADMIN 只能公告給自己部門。前台過濾用 `EXISTS` 子查詢比對使用者部門是否在目標部門列表中。`announcement_depts` 設有 `ON DELETE CASCADE`，公告刪除時自動清除。

管理頁面受眾欄位顯示：
- `scope=ALL` → 顯示「全體」
- `scope=DEPT` → 顯示目標部門名稱（如「北區、工程股」）

#### 8-14 已讀標記 upsert

使用者快速點擊同一則公告時，重複 INSERT 會因 UNIQUE 約束失敗。實作時使用：
```sql
INSERT INTO announcement_reads (announcement_id, user_id)
VALUES (:announcementId, :userId)
ON CONFLICT (announcement_id, user_id) DO NOTHING
```

#### 8-15 管理頁面搜尋與篩選

管理員公告列表提供以下篩選功能：
- **狀態篩選**：全部 / 草稿 / 已發佈 / 已過期
- **標題搜尋**：模糊比對公告標題（`LIKE '%keyword%'`）

**「已過期」為虛擬狀態**，不需要排程器更新 DB status。前端傳 `statusFilter` 參數，後端動態組合查詢條件：

| statusFilter | 後端 SQL 條件 |
|---|---|
| `ALL` | 不加 status 過濾 |
| `DRAFT` | `status = 'DRAFT'` |
| `PUBLISHED` | `status = 'PUBLISHED' AND (expire_at IS NULL OR expire_at > now())` |
| `EXPIRED` | `status = 'PUBLISHED' AND expire_at IS NOT NULL AND expire_at < now()` |

此做法與 Section 8-2（排程發佈不需排程器）一致：**純靠 SQL 查詢條件即時推導，不引入額外排程基礎設施**。
