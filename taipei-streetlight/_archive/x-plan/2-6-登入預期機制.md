## 設計規劃：登入逾時機制

### 現狀分析

| 項目 | 現況 |
|------|------|
| Access Token 過期 | 30 分鐘（YAML 設定） |
| Refresh Token 過期 | 7 天（HttpOnly cookie） |
| 401 處理 | Axios 攔截器自動 refresh → 失敗才登出 |
| 閒置偵測 | **無** |
| 系統設定表 | **無**（所有設定值都在 YAML） |

**問題**：即使使用者離開電腦 30 分鐘，access token 過期後 refresh token 仍有效（7 天），Axios 攔截器會靜默刷新 token，使用者永遠不會被登出。

### 1. 後端：系統設定表

**Flyway Migration** (`V28__system_settings.sql`)

```sql
CREATE TABLE system_settings (
    id            BIGSERIAL    PRIMARY KEY,
    tenant_id     VARCHAR(50)  NOT NULL REFERENCES tenant(tenant_id),
    setting_key   VARCHAR(100) NOT NULL,
    setting_value VARCHAR(500) NOT NULL,
    description   VARCHAR(500),
    create_time   TIMESTAMP    NOT NULL DEFAULT now(),
    update_time   TIMESTAMP    NOT NULL DEFAULT now(),
    UNIQUE (tenant_id, setting_key)
);

CREATE INDEX idx_system_settings_tenant_id ON system_settings(tenant_id);

-- 為每個既有租戶插入預設值
INSERT INTO system_settings (tenant_id, setting_key, setting_value, description)
SELECT tenant_id, 'idle_timeout_minutes', '15', '使用者閒置自動登出時間（分鐘）'
FROM tenant;
```

**多租戶設計**：每個租戶各自擁有一份設定，UNIQUE 約束為 `(tenant_id, setting_key)` 複合鍵。
Entity 遵循既有 pattern：`@Filter("tenantFilter")`、`TenantAware`、`TenantEntityListener`，Repository 繼承 `TenantScopedRepository`。

**為什麼用 generic key-value 表**：未來 2-4（密碼長度）、2-8（資安參數）等設定都可復用，且每個租戶可獨立配置。

**API 端點**

| Method | Path | 權限 | 說明 |
|--------|------|------|------|
| `GET` | `/v1/auth/system-settings/idle-timeout` | 已登入 | 取得當前租戶的閒置時間（分鐘） |
| `PUT` | `/v1/auth/system-settings/idle-timeout` | `ROLE_ADMIN` / `ROLE_SUPER_ADMIN` | 修改當前租戶的閒置時間（限 1~480 分鐘） |

因為是多租戶設計，需要從 JWT 取得 `tenantId` 才能查詢對應租戶的設定，所以 GET 改為需認證的 `/auth` 路徑。前端在登入成功、取得 token 後立即 fetch。Hibernate `@Filter` 自動依 `TenantContext` 過濾，無需手動帶 tenantId。

### 2. 前端：閒置偵測 Composable

**`useIdleTimeout.ts`** — 在 `App.vue` 掛載

```
生命週期：
1. 使用者登入成功
2. fetch idle_timeout_minutes (e.g. 15)
3. 啟動監聽：mousemove, mousedown, keydown, scroll, touchstart
4. 每次活動 → 更新 lastActivityTime（throttle 30s）
5. setInterval 每 30s 檢查 now - lastActivityTime
6. 距離超時 ≤ 2 分鐘 → 顯示倒數 Dialog
7. Dialog 按「繼續使用」→ 重置計時器
8. 倒數歸零 → POST /v1/auth/idle-logout → router.push('/login')
```

**關鍵設計決策**：

| 決策 | 方案 | 理由 |
|------|------|------|
| 事件 throttle | 30 秒 | 避免高頻事件影響效能 |
| 檢查間隔 | 30 秒 | 精度足夠，overhead 極低 |
| 警告提前量 | 2 分鐘 | 給使用者反應時間 |
| 跨分頁同步 | `BroadcastChannel` | 多個 tab 共享活動狀態，任一 tab 有操作就重置所有 tab |
| 計時器暫停 | 瀏覽器 `visibilitychange` | 切到背景時改用 `Date.now()` 差值判斷，避免 `setInterval` 被瀏覽器降頻 |

**使用者情境**（以預設 15 分鐘為例）：

```
情境 A：正常操作（使用者無感）
═══════════════════════════════
使用者持續操作（滑鼠、鍵盤、捲動）
  → 每次操作重置 lastActivityTime（throttle 30s）
  → 計時器永遠不會接近超時
  → 使用者完全無感，無任何提示

情境 B：閒置 → 警告 → 繼續使用
═══════════════════════════════
00:00  使用者停止操作（去開會、上廁所…）
       ↓ 背景靜默倒數，畫面無變化
13:00  距離超時剩 2 分鐘
       ↓
       ┌──────────────────────────────┐
       │    ⚠️ 閒置逾時警告            │
       │                              │
       │  您已閒置一段時間，           │
       │  即將自動登出。               │
       │                              │
       │         1:47                  │
       │                              │
       │      [ 繼續使用 ]             │
       │                              │
       │  ✗ 不可按 ESC / 點遮罩關閉   │
       │  ✗ 沒有 X 關閉按鈕           │
       └──────────────────────────────┘
       ↓
       使用者回來，按「繼續使用」
       → 重置計時器，Dialog 關閉
       → 回到情境 A

情境 C：閒置 → 警告 → 無回應 → 自動登出
═══════════════════════════════════════
00:00  使用者停止操作
13:00  彈出警告 Dialog（倒數 2:00）
       ↓ 使用者不在座位，無人回應
15:00  倒數歸零
       → POST /v1/auth/idle-logout（記錄審計事件）
       → 清除本地 token + 導向登入頁
       → 其他 tab 收到 BroadcastChannel 通知，同步登出
```

**設計模式選擇**：採用模式 2（閒置偵測），不在 Header 顯示固定倒數計時器。
- 模式 1（固定倒數）：適合網路銀行等每筆操作涉及金錢的場景，對 IoT 管理平台過度打擾
- 模式 2（閒置偵測）：正常操作完全無感，只在真正閒置時才彈出 Dialog 警告 ✅
- 模式 3（雙計時器）：疊加絕對 session 上限，適合更高安全場域，目前不需要

### 3. 與現有 JWT 機制的配合

```
閒置時間 < Access Token 過期時間 (30min)
     ↓
使用者閒置 15 分鐘 → 前端主動登出（不等 token 過期）
使用者持續操作 → 30 分鐘 access token 過期
                → Axios 攔截器透明 refresh
                → 使用者無感繼續操作
```

**建議**：`idle_timeout_minutes` 預設值 **15 分鐘**，小於 access token 的 30 分鐘。這樣閒置登出永遠先觸發，不會出現「token 過期但還沒登出」的不一致狀態。

若管理員將閒置時間設為 > 30 分鐘，則 access token 會先過期但 refresh token 會自動續期，使用者仍在閒置時間到達後被登出，邏輯依然正確。

### 4. 審計事件設計

閒置逾時登出需記錄為獨立的審計事件，與使用者主動登出區分。

**新增 `AuditEventType.IDLE_TIMEOUT_LOGOUT`**

| 事件 | 觸發者 | AuditEventType | AuditCategory |
|------|--------|----------------|---------------|
| 使用者按「登出」 | 使用者主動 | `LOGOUT` | `USER_AUTH` |
| 閒置逾時被踢出 | 系統自動 | `IDLE_TIMEOUT_LOGOUT` | `USER_AUTH` |

**設計理由**：
- 與現有風格一致（`LOGIN_SUCCESS` / `LOGIN_FAIL` 分開，而非同一個 `LOGIN` 加描述區分）
- Audit History 頁面可直接用 eventType 過濾，查看所有閒置逾時事件
- 資安稽核可分析：某帳號頻繁閒置逾時 → 帳號共用或未妥善管理

**API 變更**：新增獨立 endpoint，因 `@AuditEvent` 註解是編譯期綁定，無法動態切換事件類型

```
POST /v1/auth/logout          → @AuditEvent(LOGOUT)
POST /v1/auth/idle-logout     → @AuditEvent(IDLE_TIMEOUT_LOGOUT)
```

兩個 endpoint 共用私有 `doLogout()` 方法（清除 refresh token cookie），僅審計事件類型不同。

### 5. 邊界情境處理

#### 5-1 新租戶缺少預設設定

Flyway migration 只為既有租戶插入 `idle_timeout_minutes`，之後新建的租戶不會有這筆資料。

**方案**：雙重保障
- `TenantService` 建立租戶時，同步 seed 所有 `system_settings` 預設值
- `SystemSettingService.getIdleTimeout()` 查無資料時回傳 hardcoded default（15 分鐘），避免遺漏

#### 5-2 選單與權限

管理後台設定頁面需要對應的 menu 與 permission，一併放在 V28 migration：
- `menu`：系統設定頁（掛在管理選單下）
- `permission`：`SYSTEM_SETTINGS_MANAGE`
- `role_permissions`：綁定到 ADMIN / SUPER_ADMIN 角色

#### 5-3 前端 fallback 預設值

若 `GET /idle-timeout` 失敗（網路異常、設定遺失），composable 使用 hardcoded fallback **15 分鐘**，避免閒置偵測完全失效。

```typescript
const DEFAULT_IDLE_TIMEOUT_MINUTES = 15
```

#### 5-4 多 tab 登出只呼叫一次 API

BroadcastChannel 同步活動狀態，但登出時需避免重複：
- **觸發逾時的 tab**：呼叫 `POST /logout?reason=idle_timeout`，然後透過 BroadcastChannel 廣播 `{ type: 'idle_logout' }`
- **其他 tab**：收到訊息後，只做本地清除（`authStore.clearAuth()`）+ 導向登入頁，**不再呼叫 logout API**
- 避免多個 tab 同時打 logout API 產生重複審計事件

#### 5-5 Token 已過期時的登出處理

若 `idle_timeout > 30min`，閒置逾時觸發時 access token 可能已過期：

```
閒置逾時觸發
  → POST /logout 回 401
  → Axios 攔截器嘗試 refresh
  → refresh 成功 → 重發 logout → 正常記錄審計事件 ✅
  → refresh 失敗 → 前端仍強制清除本地狀態 + 導向登入 ✅
                  （審計事件遺失，可接受）
```

前端 logout 流程採 **fire-and-forget**：不管 API 成功或失敗，都執行 `clearAuth()` + `router.push('/login')`。
